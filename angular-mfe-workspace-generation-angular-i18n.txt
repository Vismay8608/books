ANGULAR 21+ PRODUCTION-GRADE MICROFRONTEND WORKSPACE GENERATION SPEC
====================================================================

Purpose
-------
Use this TXT as a ready-to-paste generation prompt / implementation blueprint for creating a latest stable Angular enterprise workspace with:

- Angular latest stable production setup; as of 2026-05-22 use Angular 21.x stable, not Angular 22 RC.
- Multi-project Angular workspace.
- Webpack Module Federation microfrontend architecture.
- Host shell application.
- Admin remote microfrontend.
- Admin Asset Management feature with Create, Edit, View, Search and Manage Asset components.
- Production build configurations.
- Multilingual setup using Angular built-in i18n / @angular/localize. No third-party translation provider.
- Guards, interceptors, services, utilities, theme components and layouts.
- Shared utilities managed inside the shell application, not as a separate Angular library.
- Strict TypeScript, SCSS, lazy loading, environment configuration and CI-ready scripts.

Recommended versions
--------------------
Node.js: active LTS compatible with Angular 21
Package manager: npm or pnpm
Angular CLI: latest Angular 21 stable
Microfrontend package: @angular-architects/module-federation compatible with Angular 21
Translation: Angular built-in i18n using @angular/localize. Do not use Transloco, ngx-translate or any third-party translation provider.

============================================================
1. CREATE WORKSPACE AND PROJECTS
============================================================

Run these commands from an empty folder:

```bash
npm install -g @angular/cli@latest

ng new enterprise-mfe-workspace \
  --create-application=false \
  --style=scss \
  --strict=true \
  --package-manager=npm

cd enterprise-mfe-workspace

ng generate application shell \
  --routing=true \
  --style=scss \
  --standalone=true

ng generate application admin \
  --routing=true \
  --style=scss \
  --standalone=true

ng generate library shared-ui
ng generate library shared-data-access
```

Install production dependencies:

```bash
npm install @angular-architects/module-federation --save-dev
ng add @angular/localize
npm install uuid
npm install --save-dev webpack webpack-cli webpack-merge
```

Add Module Federation:

```bash
ng add @angular-architects/module-federation --project shell --type host --port 4200
ng add @angular-architects/module-federation --project admin --type remote --port 4201
```

If the schematic asks for standalone routing support, answer YES.

4. SHELL-OWNED SHARED UTILITIES, NO SHARED-UTIL LIBRARY
============================================================

Use this structure instead of creating a `shared-util` Angular library:

```text
projects/shell/src/app/shared/
  constants/
    app.constants.ts
    route.constants.ts
    storage-key.constants.ts
  utilities/
    date.util.ts
    string.util.ts
    object.util.ts
    route.util.ts
    permission.util.ts
    validation.util.ts
  validators/
    common.validators.ts
  types/
    common.types.ts
  index.ts
```

Rules:
- Shell can import from `@shell-shared` or relative shell paths.
- Admin remote must not import from `projects/shell/src/app/shared`.
- Keep remote deployability independent.
- If a utility is truly cross-application, either duplicate the small pure function in the remote or promote it later to a proper shared package/library.
- Keep shell shared utilities pure and framework-light where possible.

Recommended path alias in `tsconfig.base.json`:

```json
{
  "compilerOptions": {
    "paths": {
      "@shell-shared": ["projects/shell/src/app/shared/index.ts"],
      "@shell-shared/*": ["projects/shell/src/app/shared/*"]
    }
  }
}
```

Sample `projects/shell/src/app/shared/index.ts`:

```ts
export * from './constants/app.constants';
export * from './constants/route.constants';
export * from './constants/storage-key.constants';
export * from './utilities/date.util';
export * from './utilities/string.util';
export * from './utilities/object.util';
export * from './utilities/route.util';
export * from './utilities/permission.util';
export * from './utilities/validation.util';
export * from './validators/common.validators';
export * from './types/common.types';
```

```ts
export function hasAnyRole(userRoles: readonly string[], requiredRoles: readonly string[]): boolean {
  if (!requiredRoles.length) return true;
  return requiredRoles.some(role => userRoles.includes(role));
}

export function hasPermission(userPermissions: readonly string[], permission: string): boolean {
  return userPermissions.includes(permission);
}
```

`projects/shell/src/app/shared/utilities/string.util.ts`

```ts
export function isBlank(value: string | null | undefined): boolean {
  return !value || value.trim().length === 0;
}

export function toTitleCase(value: string): string {
  return value
    .toLowerCase()
    .split(' ')
    .filter(Boolean)
    .map(word => word.charAt(0).toUpperCase() + word.slice(1))
    .join(' ');
}
```

`projects/shell/src/app/shared/validators/common.validators.ts`

```ts
import { AbstractControl, ValidationErrors, ValidatorFn } from '@angular/forms';

export function noWhitespaceValidator(): ValidatorFn {
  return (control: AbstractControl): ValidationErrors | null => {
    const value = control.value as string | null | undefined;
    return value && value.trim().length === 0 ? { whitespace: true } : null;
  };
}
```

9. SHELL ROUTING
============================================================

projects/shell/src/app/app.routes.ts

```ts
import { Routes } from '@angular/router';
import { loadRemoteModule } from '@angular-architects/module-federation';
import { authGuard } from './core/guards/auth.guard';
import { roleGuard } from './core/guards/role.guard';
import { mfeAvailableGuard } from './core/guards/mfe-available.guard';

export const routes: Routes = [
  {
    path: '',
    loadComponent: () =>
      import('./core/theme/layouts/shell-layout/shell-layout.component')
        .then(m => m.ShellLayoutComponent),
    canActivate: [authGuard],
    children: [
      {
        path: '',
        pathMatch: 'full',
        redirectTo: 'dashboard'
      },
      {
        path: 'dashboard',
        loadComponent: () =>
          import('./modules/dashboard/dashboard.component')
            .then(m => m.DashboardComponent)
      },
      {
        path: 'admin',
        canMatch: [mfeAvailableGuard, roleGuard],
        data: {
          remoteName: 'admin',
          roles: ['ADMIN', 'ASSET_MANAGER']
        },
        loadChildren: () =>
          loadRemoteModule({
            type: 'module',
            remoteEntry: 'http://localhost:4201/remoteEntry.js',
            exposedModule: './Routes'
          }).then(m => m.ADMIN_ROUTES)
      }
    ]
  },
  {
    path: 'auth',
    loadChildren: () => import('./modules/auth/auth.routes').then(m => m.AUTH_ROUTES)
  },
  {
    path: 'forbidden',
    loadComponent: () => import('./modules/errors/forbidden/forbidden.component').then(m => m.ForbiddenComponent)
  },
  {
    path: 'remote-unavailable',
    loadComponent: () => import('./modules/errors/remote-unavailable/remote-unavailable.component').then(m => m.RemoteUnavailableComponent)
  },
  {
    path: '**',
    loadComponent: () => import('./modules/errors/not-found/not-found.component').then(m => m.NotFoundComponent)
  }
];
```

For production, replace hard-coded remoteEntry with a dynamic manifest resolver.

```ts
import { HttpErrorResponse, HttpInterceptorFn } from '@angular/common/http';
import { inject } from '@angular/core';
import { Router } from '@angular/router';
import { catchError, throwError } from 'rxjs';
import { NotificationService } from '../services/notification.service';

export const errorInterceptor: HttpInterceptorFn = (req, next) => {
  const router = inject(Router);
  const notification = inject(NotificationService);

  return next(req).pipe(
    catchError((error: HttpErrorResponse) => {
      if (error.status === 401) {
        router.navigate(['/auth/login']);
      } else if (error.status === 403) {
        router.navigate(['/forbidden']);
      } else {
        notification.error(error.error?.message || 'Something went wrong');
      }

      return throwError(() => error);
    })
  );
};
```

```ts
export interface CurrentUser {

  displayName: string;
  roles: string[];

@Injectable({ providedIn: 'root' })

  readonly user = this.userState.asReadonly();

    return !!this.getAccessToken();

    return roles.some(role => userRoles.includes(role));
  }

  hasPermission(permission: string): boolean {
    return this.userState()?.permissions.includes(permission) ?? false;

  logout(): void {
    localStorage.removeItem('access_token');
    this.userState.set(null);
  }
}
```

```ts
import { Injectable } from '@angular/core';

export type SupportedLocale = 'en' | 'hi';

@Injectable({ providedIn: 'root' })
export class LocaleService {
  private readonly storageKey = 'selected_language';
  private readonly supportedLocales: SupportedLocale[] = ['en', 'hi'];

  initializeLocale(): void {
    const locale = this.getCurrentLocale();
    localStorage.setItem(this.storageKey, locale);
    document.documentElement.lang = locale;
  }

  getCurrentLocale(): SupportedLocale {
    const pathLocale = window.location.pathname.split('/').filter(Boolean)[0] as SupportedLocale | undefined;
    if (pathLocale && this.supportedLocales.includes(pathLocale)) return pathLocale;

    const savedLocale = localStorage.getItem(this.storageKey) as SupportedLocale | null;
    if (savedLocale && this.supportedLocales.includes(savedLocale)) return savedLocale;

    return 'en';
  }

  changeLocale(locale: SupportedLocale): void {
    if (!this.supportedLocales.includes(locale)) return;
    localStorage.setItem(this.storageKey, locale);

    const segments = window.location.pathname.split('/').filter(Boolean);
    if (segments[0] === 'en' || segments[0] === 'hi') {
      segments[0] = locale;
    } else {
      segments.unshift(locale);
    }

    window.location.href = '/' + segments.join('/') + window.location.search + window.location.hash;
  }
}
```

```ts
import { HttpClient, HttpParams } from '@angular/common/http';
import { inject, Injectable } from '@angular/core';
import { Observable } from 'rxjs';
import { Asset } from '../models/asset.model';
import { AssetFilter } from '../models/asset-filter.model';
import { AssetFormModel } from '../models/asset-form.model';

export interface PageResult<T> {
  items: T[];
  page: number;
  size: number;
  total: number;
}

@Injectable({ providedIn: 'root' })
export class AssetApiService {
  private readonly http = inject(HttpClient);
  private readonly baseUrl = '/api/admin/assets';

  search(filter: AssetFilter): Observable<PageResult<Asset>> {
    let params = new HttpParams()
      .set('page', filter.page)
      .set('size', filter.size);

    if (filter.searchText) params = params.set('searchText', filter.searchText);
    if (filter.category) params = params.set('category', filter.category);
    if (filter.location) params = params.set('location', filter.location);
    if (filter.status) params = params.set('status', filter.status);

    return this.http.get<PageResult<Asset>>(this.baseUrl, { params });
  }

  getById(assetId: string): Observable<Asset> {
    return this.http.get<Asset>(`${this.baseUrl}/${assetId}`);
  }

  create(payload: AssetFormModel): Observable<Asset> {
    return this.http.post<Asset>(this.baseUrl, payload);
  }

  update(assetId: string, payload: AssetFormModel): Observable<Asset> {
    return this.http.put<Asset>(`${this.baseUrl}/${assetId}`, payload);
  }

  delete(assetId: string): Observable<void> {
    return this.http.delete<void>(`${this.baseUrl}/${assetId}`);
  }
}
```

============================================================

Use Angular built-in internationalization with `@angular/localize`.
Do not use Transloco, ngx-translate or any other third-party translation provider.

Important behavior:
- Angular built-in i18n is primarily compile-time localization.
- Text is translated using Angular i18n markers in templates or `$localize` in TypeScript.
- Production usually creates one build per locale, for example `/en/` and `/hi/`.
- Language switching navigates to the locale-specific deployed app path instead of changing JSON at runtime.

20. PRODUCTION-GRADE MICROFRONTEND CHECKLIST
============================================================

Architecture:
- Shell owns global authentication, navigation, layout and remote discovery.
- Remote owns its feature routes, feature services and feature UI.
- Shared libraries contain only stable contracts and reusable UI. Shell-only utilities stay inside `projects/shell/src/app/shared`.
- Avoid tight coupling from remote to shell internals.

Security:
- Validate authentication and authorization on backend also.
- Frontend guards are UX controls, not security boundaries.
- Use HTTPS for remoteEntry files in production.
- Use CSP headers.
- Restrict CORS to trusted origins.
- Add versioned remote deployment paths, for example `/mfe/admin/v1/remoteEntry.js`.

Performance:
- Lazy load all feature routes.
- Keep shared dependencies singleton.
- Use production budgets.
- Avoid loading all remotes at startup unless required.
- Prefer dynamic remote manifest loading.

Reliability:
- Add remote-unavailable route/page.
- Log remote load failures.
- Allow disabling a remote from remote-manifest.json.
- Keep shell backwards-compatible with at least one previous remote version.

Testing:
- Unit tests for guards, interceptors, services and asset forms.
